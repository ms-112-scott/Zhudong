# 開發技術文件

本文件供接手或維護本專案的工程師參考，涵蓋整體架構、資料流、技術棧、各模組細節、發佈機制與已知問題。

## 一、專案概要

本專案為 NYCU 竹東農再計畫網站。核心是一個以 D3 力導向圖呈現的知識圖譜。資料以 Google Sheets 作為內容管理來源，透過 Google Apps Script 將整理後的 JSON 推送到 GitHub repo，前端在執行階段直接 fetch GitHub 的 raw 檔案進行渲染。

設計上採用「靜態 JSON 加 GitHub raw CDN」取代即時後端 API，目的是省去伺服器成本與維運，代價是資料更新有約五分鐘的快取延遲。

## 三、整體架構與資料流

資料流分為兩條獨立軌道。

資料軌：

```
圖譜試算表　─(Graph.gs)→　master 的 graph.json
圖片試算表　─(Image.gs)→　master 的 images.json
Google Drive 圖片資料夾　─(被 Image.gs 掃描)
master 的 *.json　─(raw CDN fetch)→　網站
```

程式碼軌：

```
master 的 src/ 原始碼　─(npm run deploy)→　gh-pages branch　─(GitHub Pages)→　網站
```

兩條軌道的重點：

- 資料軌：Sheets 經 Apps Script 推到 master 的 JSON，前端執行階段 fetch raw 檔。修改資料不需要重新部署。
- 程式碼軌：master 的 src 經建置後推到 gh-pages，由 Pages 提供服務。修改程式碼才需要部署。

### Branch 配置

| Branch | 內容 | 由誰寫入 |
|--------|------|----------|
| master（預設分支） | 原始碼與 src/data 下的 JSON | 開發者 push；Apps Script 透過 GitHub Contents API PUT，因未指定 branch，預設打到預設分支 master |
| gh-pages | 建置後的 dist 靜態檔 | npm run deploy 自動推送 |

GitHub Pages 設定：Settings 進入 Pages，Source 選 Deploy from a branch，分支選 gh-pages，目錄選 root。

## 四、後端：Google Apps Script

位於 appscript 目錄。兩支腳本各自綁定一本試算表，完全獨立，各有自己的 onOpen 簡單觸發器與 GITHUB_TOKEN 指令碼屬性。

### Graph.gs（圖譜試算表）

設定常數 GRAPH_CONFIG，包含 REPO_OWNER、REPO_NAME 與 FILE_PATH，後者為 src/data/graph.json。

選單「圖譜工具」：

- runGraphCheckWithUI：只執行 processGraphData，做檢查、上色與更新 missing_node，不推送。
- runGraphSyncWithUI：執行檢查並呼叫 pushGraphToGitHub。

processGraphData 核心流程：

1. 建立顏色表（nodes_color 與 links_color）與圖例。
2. 第一階段掃描 link_ 分頁，記錄每個 ID 出現位置，建立自動節點（暫定 group 為自動增加），收集 links。
3. 第二階段掃描 nodes_ 分頁，覆蓋真實資料，合併同 ID 的 Info，檢查 Group 一致性並記錄衝突。
4. 第三階段批次上色，紅色代表衝突，黃色代表仍為自動增加（即缺漏節點）。
5. 更新 missing_node 分頁，回傳 result（含 nodes、links、legend）與 summary。

pushGraphToGitHub：先 GET 取得現有檔案的 sha 與舊內容，與新內容字串比對，相同則跳過推送，不同才 PUT，commit 訊息為 Auto-sync (content changed): graph.json。

doGet：保留的 Web App 入口，向下相容，目前已不使用，前端改讀 raw 檔。

### Image.gs（圖片試算表）

設定常數 SYNC_CONFIG，包含 IMAGE_FOLDER_ID（Drive 資料夾）、repo 設定、FILE_PATH 為 src/data/images.json，以及 MISSING_SHEET 為 missing_img。

選單「圖片同步」呼叫 runSyncImagesWithUI，進而執行 syncImagesToGitHub。

syncImagesToGitHub 流程：

1. collectImages 遞迴掃描 Drive 資料夾與子層，取得每張圖的 name、id、path。
2. 走訪 image_ 分頁，A 欄為 node_id（輸出用），B 欄為 file_name（當前綴用），C 欄為 info。
3. 對每個前綴，找出檔名以前綴開頭的圖，收進 finalJson，每筆包含 img_id、node_id、file_name、info、sheet_source。
4. 問題分三類寫進 missing_img：找不到圖、重複檔名（不同資料夾同名）、孤兒圖檔（沒被任何前綴配到）。
5. finalJson 為空則中止以防呆，否則呼叫 pushImagesToGitHub，同樣以 sha 比對，有變動才 PUT。

### Token

存放於 Apps Script 的專案設定中的指令碼屬性，鍵名為 GITHUB_TOKEN，使用 classic PAT，scope 只需 repo。詳見 appscript/TOKEN_設定說明.md。

## 五、前端結構

### 進入點與路由

```
main.jsx → App.jsx
  DataProvider（images.json 的 context）
    HashRouter
      Layout（Navbar、Footer 外框）
        GraphProvider（graph.json 的 context，包住整個 Routes，全頁共享）
          Routes
            /                       HomePage
            /KnowledgeGraphPage     KnowledgeGraphPage
            /education              EducationPage
            /workshop               WorkshopPage
            /gallery                GalleryPage
```

採用 HashRouter，URL 帶有 #，因為 GitHub Pages 對未知路徑會回 404，hash 路由可繞過此限制。vite.config.js 的 base 設為 /Zhudong，對應 repo 名稱的子路徑。

### 資料層（兩個 Context）

| Context | 檔案 | 抓取對象 | 提供內容 |
|---------|------|----------|----------|
| DataProvider | src/context/DataContext.jsx | images.json | images、loading |
| GraphProvider | src/components/KnowledgeGraph/GraphProvider.jsx | graph.json | graphData（清洗後）、isLoading、error |

兩者皆 fetch raw.githubusercontent.com 上 master 分支的 JSON，URL 後接 ?t=Date.now() 做 cache-busting，詳見第七節。

GraphProvider 在 fetch 後做資料清洗：將 node 的 id 強制轉為字串、以 Set 建立合法 ID 索引、過濾掉端點不存在的 links。

### 知識圖譜核心

```
KnowledgeGraphPage.jsx
  useKnowledgeGraph(graphData, containerRef)  // src/hooks/useKnowledgeGraph.jsx
    年份邊界計算（globalYearRange）
    filteredData（依 filterYearRange 過濾節點與連線，useMemo）
    new ForceGraph(container, { onNodeClick, onBackgroundClick })  // D3 class
    高亮同步（selectedNode 變動時呼叫 highlightNode 或 resetHighlight）
    actions：zoomToNode、zoomToFit、highlightGroup、setSelectedNode、setFilterYearRange 等
```

D3 封裝位於 src/components/KnowledgeGraph/Graph：

- ForceGraph.js：主 class，處理 force simulation、zoom、drag 與 render。
- GraphRenderers.js：繪圖。
- GraphInteractions.js：點擊、hover、drag 等互動。
- GraphTimeline.js：時間軸相關。
- GraphUtils.js：工具函式。

時間軸過濾邏輯（useKnowledgeGraph 的過濾段）：節點若無 start_year 或非數字，視為永久顯示；否則以節點的起訖年份與 slider 區間做交集判斷。

UI 元件位於 src/components/KnowledgeGraph：

- InfoCard.jsx：右側節點資訊卡，分四區，分別為標題、描述、關聯標籤（依鄰居 group 分組）與媒體。
- InfoCard/MediaSection.jsx：依 node.id 從 images 過濾出 nodeImages，呈現橫向捲動畫廊。圖片來源為 drive.google.com/thumbnail，帶 img_id 與尺寸參數。內含除錯 log，點節點時印出對應圖片數與檔名。
- GraphSearch.jsx、GraphToolbar.jsx、GraphLegend.jsx、TimelineSlider.jsx、TimelineOverlay.jsx。
- FilteredGraphLogger.jsx：無介面除錯元件，追蹤過濾後的資料流。

### 其他頁面

HomePage、EducationPage、WorkshopPage、GalleryPage 與 components 下的 HomePage、GalleryPage、global、helper 為內容與版面元件。Gallery 使用 DataProvider 提供的 images。

## 六、資料格式

graph.json：

```jsonc
{
  "nodes": [
    {
      "id": "A01", "group": "人物", "info": "說明文字",
      "start_year": 1900, "end_year": null,
      "lon": null, "lat": null, "col": "#CCCCCC"
    }
  ],
  "links": [
    { "source": "A01", "target": "B02", "group": "關係", "label": "", "info": "", "color": "#999999" }
  ],
  "legend": [ { "label": "人物", "color": "#..." } ]
}
```

images.json：

```jsonc
[
  { "img_id": "<Drive file id>", "node_id": "A01", "file_name": "A01_1.jpeg", "info": "", "sheet_source": "image_Main" }
]
```

前端 MediaSection 以 String(img.node_id) 等於 String(node.id) 配圖。

## 七、快取與發佈延遲

### 兩層快取

1. 瀏覽器快取：開發時可在 F12 的 Network 勾選 Disable cache，或按 Ctrl+Shift+R。
2. GitHub raw CDN 快取：raw.githubusercontent.com 對同一檔案有約五分鐘的 server-side TTL。

### cache-busting

GraphProvider.jsx 與 DataContext.jsx 的 fetch 都加上 ?t=Date.now()，每次載入是不同 URL，可繞過邊緣快取。

已知限制：?t= 只繞邊緣快取，raw 的 origin TTL（約五分鐘）仍可能回舊內容。實測即使 t 每次變動，master 已更新後 raw 仍可能落後數分鐘。這不是程式錯誤，而是 raw CDN 的機制，只能等待。

排查順序，由後往前：

1. master blob 是否為新（github.com 下 blob/master 的 JSON）。master 為即時，最權威。
2. raw URL 是否為新（raw.githubusercontent.com 下 master 的檔，帶隨機 t）。此處落後 master 約五分鐘。
3. 若兩者都新但畫面仍舊，則為前端 state 或瀏覽器快取問題。

### 如需零延遲

唯一方式是不走 raw CDN，改用 GitHub Contents API（api.github.com 的 contents 端點，回傳 base64，即時但有 rate limit，且需留意 token 曝露），或改用自架、雲端函式。目前架構是刻意以成本換取五分鐘延遲。

## 八、開發與部署

本機開發：

```bash
npm install
npm run dev        # Vite dev server，修改 src 即時生效
npm run lint       # ESLint
npm run build      # 產出 dist
npm run preview    # 預覽 build 結果
```

本機 dev 仍直接 fetch 線上 raw JSON，因此本機也受五分鐘 CDN 延遲影響，但前端程式碼改動為即時生效。

部署：

```bash
npm run deploy     # 等同 predeploy(build) 加 npx gh-pages -d dist
```

此指令把 dist 推到 gh-pages 分支。GitHub Pages 建置完成約需一到十分鐘上線，可於 github.com 的 deployments 頁面確認。

修改前端 fetch 邏輯（例如 cache-busting）後，務必重新執行 npm run deploy，否則瀏覽器仍會跑舊的 bundle，即使資料已更新。

.github 目錄目前為空，沒有 GitHub Actions，部署為手動執行 npm run deploy。若要做 CI 自動部署，可在此加入 workflow。

## 九、已知問題與注意事項

- ID 一致性是整個系統的關鍵。圖譜的 node ID、link 的 source 與 target、圖片的 node_id 三處必須字串完全一致，相差一個空格、繁簡或大小寫就會斷開。Apps Script 端以黃色、紅色標記與 missing 分頁輔助。
- raw CDN 的 URL 寫死 master。GraphProvider.jsx 與 DataContext.jsx 寫死分支名稱，若更改預設分支需同步修改這兩處。
- Apps Script 的 PUT 未指定 branch，依賴 repo 的預設分支。更改預設分支會影響推送目標。
- Drive 圖片權限。drive.google.com/thumbnail 需檔案設為知道連結的人可檢視，否則前端顯示 Image restricted。
- 重複檔名。不同子資料夾的同名圖會在 images.json 造成 img_id 衝突，Apps Script 會列入 missing_img 提醒。
- HashRouter。所有路徑帶 #，分享連結時注意格式。
- cache-busting 的流量代價。?t=Date.now() 讓每位訪客每次載入都重抓 JSON，流量上升。正式環境若要省流量，可改為固定或版本化的 cache key，並接受延遲。
